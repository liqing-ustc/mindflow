端到端导航方向讨论

Mar 27, 20266 min read

  • navigation
  • spatial-intelligence
  • VLA

核心议题

  1. 端到端导航的可行性与方向
  2. VLA 框架中 V/L/A 的能力差距分析

Discussion Notes

1. 人机对话与导航指令

  • 导航场景下不需要完全开放的自然语言，指令格式可以相对统一（如”往前移动多少”、“在哪右拐”）
  • 关键在于内容的泛化性，而非语言形式的多样性
  • 给模型明确的输入格式后，输出端可以灵活：基于 waypoint + local planner，或直接端到端输出均可

2. Sim-to-Real Gap 与任务定义问题

  • 现状：让机器人导航到某个会议室门口这样的基础任务都做不到
  • 仿真中分数刷很高，但即使没有 sim-to-real gap，真机上也无法直接用
  • 根因：任务定义本身有问题——这是一个绕不过去的 gap
  • 如果要让机器人在家庭/办公场景中完成复杂任务，场景级导航是 long-term 必须解决的问题

3. 端到端 vs. SLAM（重要立场）

  • 倾向端到端方式：更容易 scale，未来更有潜力
  • SLAM 偏工程化：在受限场景下能调通，但长远需要用 model 解决导航+操作问题
  • 理想形态：一个视频流输入 → 模型内部自行处理 → 输出动作，不需要外挂很多模块

4. 导航动作空间的量化思考

  • 方向量化：每 10° 一个 token → 36 个离散动作
  • 每次移动固定距离（如 5cm）
  • 导航的动作空间本身很有限（~36），不追求动作泛化，追求场景泛化
  • 每个离散动作可视为一种 skill
  • 头顶装一个摄像头 → 结合多模态大模型能力 → 导航变成一个 proxy task，能把空间感知做强

5. 实时避障与模型内化

  • 希望将避障也做到 model 内部，而非依赖传统 SLAM 避障（“有点像靠撞的”）
  • Jason 观点：避障不应是难问题，大模型（如 Qwen-VL）稍微训一训就应该能激发出避障能力
  • 与 manipulation 不同，导航+避障更适合端到端学习

6. Long Context 与视频流处理（技术瓶颈）

  • 长上下文建模可以将空间信息 encode 进去 → 支持长周期任务（“先去左边做X，再去右边做Y，再回来”）
  • 问题：当前开源多模态模型（如千问 VL）处理长视频能力不足
    • Gemini 在长视频处理上较强，开源模型普遍不行
    • 原因：暴力做 long context 不可行，需要关键帧提取/替换机制
    • 可以不做在模型内部，外挂机制也可行
  • 视频流的特殊性：帧间变化不均匀（平稳行走时变化小，转向时变化大），固定 FPS 抽帧不实用
  • 这个任务可以反过来 drive 多模态大模型的视频流理解能力

7. Spatial Memory 归属（延续上次讨论）

  • 视频流形成的 spatial memory 放在大脑侧（云端更大模型）
  • 可参考 Mimo-v2-omni 的效果，但不要太依赖——太大了
  • 整体可行：用更大模型跑在云端处理 spatial memory

8. VLA 框架中的能力差距分析（重要洞察）

L 明显超过 V 和 A

维度现状评估
L（语言推理/规划）已经非常强，能支撑当前考虑的任务✅ 基本解决
V（视觉/空间理解）停留在物体识别/图像级别；从多视角图片/视频流中 reconstruct spatial intelligence 仍欠缺❌ 核心瓶颈
A（动作执行）稳定性和泛化性不如 L⚠️ 需要提升

结论：重点应放在：

  1. 解决 V — 空间理解、感知、从视觉重建空间智能
  2. 解决 A — 动作的稳定性和泛化性
